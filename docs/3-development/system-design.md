## 1. Architecture Model: White-Label Fork Deployment

### 1.1 Ownership & Licensing

```
Keith (IP Owner)
└── Master GitHub Repository
    ├── Full platform codebase
    ├── License: proprietary (usage rights granted per licensee)
    └── Updates: licensees pull from upstream as needed
```

This is a **software licensing model**, not a SaaS model. Keith licenses the codebase to other businesses (licensees). Each licensee deploys and operates their own independent instance.

### 1.2 Two-Layer Tenant Structure

```
Keith's GitHub (master)
        ↓ fork + license agreement
┌─────────────────────────────────┐    ┌─────────────────────────────────┐
│  Licensee A: "SchoolMerch.org"  │    │  Licensee B: "TeamGear.com"     │
│  ├── Own branding & domain      │    │  ├── Own branding & domain      │
│  ├── Own DB & infrastructure    │    │  ├── Own DB & infrastructure    │
│  └── Tenants (organizations):   │    │  └── Tenants (organizations):   │
│      ├── Lincoln Elementary PTA │    │      ├── FC Downtown Soccer     │
│      ├── Jefferson Middle School│    │      ├── Lions FC               │
│      └── Regional Arts Council  │    │      └── District Tournament    │
└─────────────────────────────────┘    └─────────────────────────────────┘
```

- **Platform level**: One fork per licensee. Fully isolated (DB, infra, brand).
- **Organization level**: Multiple organizations (PTAs, teams, clubs) within each licensee's deployment.

### 1.3 White-Label Configuration

All brand-specific values live in `platform.config.ts` and environment variables. **Zero brand data is hardcoded in the codebase.**

## 3. Setup Wizard Design

### 3.1 UX Philosophy

> "The wizard holds the user's hand through every step. External services (Stripe, Printful, etc.) open in a new window — the wizard tells you exactly what to do and where to click, so even a non-technical user can complete setup."

- Every step has a **"Why we need this"** explanation
- External service links **open in a new tab** with step-by-step guidance shown alongside
- Each API key field has a **"How to get this"** inline guide (expandable)
- **Connection test** runs automatically after each key is entered
- Progress is **saved between sessions** — close and resume anytime
- The wizard is **only accessible before setup is complete**, then permanently hidden

## 6. Multi-Tenancy Security

### 6.1 Strategy

Since Turso (libSQL) does not support PostgreSQL-style Row Level Security, all org-data isolation is enforced at the **application layer** via a shared middleware function.

### 6.2 requireOrgAccess() Middleware

```typescript
// lib/middleware/requireOrgAccess.ts
export async function requireOrgAccess(
  userId: string,
  orgId: string,
  minRole: "buyer" | "member" | "student" | "admin" = "member"
): Promise<OrgMember> {
  const membership = await db.query.orgMembers.findFirst({
    where: and(
      eq(orgMembers.userId, userId),
      eq(orgMembers.orgId, orgId)
    )
  })

  if (!membership) throw new ForbiddenError("Not a member of this org")
  if (!hasRole(membership.role, minRole)) throw new ForbiddenError("Insufficient role")

  return membership
}
```

**Rules:**
- Called at the **top of every API route** that touches org-scoped data
- Never trust org context from the client — always re-validate server-side
- Tested independently to confirm no route can bypass it

### 6.3 Authentication

| Rule | Detail |
|---|---|
| Primary auth | Google OAuth (via Better Auth) |
| Secondary auth | Magic link (passwordless email) |
| Session | JWT in httpOnly cookie |
| Role enforcement | Server-side only — never trust client-sent role |
| Public pages | Campaign order pages require no auth |

## 7. Key Application Flows

### 7.1 Campaign Creation (4 Steps)

```
Step 1: Design Studio
  → Upload logo (R2) or generate AI design (OpenAI → remove.bg → R2)
  → Live mockup via Printful mockup API
  → No auth required yet (progressive registration)

Step 2: Products & Pricing
  → Select preset pack → fetch Printful base costs
  → Set retail price → live margin calculator
  → Set fundraising goal + deadline
  → Contextual tips displayed inline

Step 3: Bank Account (Stripe Connect)
  → Stripe Express embedded onboarding
  → Required before publish
  → KYC handled by Stripe

Step 4: Publish
  → Preview campaign page
  → Shareable URL generated (slug)
  → "Go Live" → campaign status: active
```

### 7.2 Order Flow (Buyer)

```
Buyer visits campaign URL (no auth required)
  → Selects product + size + quantity
  → Stripe Checkout (guest checkout supported)
  → Payment confirmed → webhook: payment_intent.succeeded
  → Printful order auto-submitted
  → Confirmation email via Resend
  → Shipping notification when dispatched
```

### 7.3 Stripe Webhook Handling

```
payment_intent.succeeded
  → Verify webhook signature (Stripe secret)
  → Create Order record (status: paid)
  → Submit to Printful API
  → Send confirmation email

account.updated (Stripe Connect)
  → Update org.stripe_onboarding_complete flag
```

### 7.4 AI Design Generation + Printful Mockup

**決定事項 (2026-06-01):**
- `remove.bg` は使用しない。`gpt-image-1` で `background: "transparent"` を指定すれば透過PNG が直接生成できる
- プログレッシブ登録ゲート（1回無料→登録）は今回スコープ外。キャンペーン作成はログイン必須のため不要
- モックアップは Printful Mockup Generator API を使用（Unisex T-Shirt をデフォルト表示）

**AI デザイン生成フロー:**
```
管理者がテキストプロンプトを入力（デザインステップ）
  → サーバーサイドで著作権フィルター（Disney/Marvel/NFL等を拒否）
  → /api/ai-design → OpenAI gpt-image-1 (background: "transparent", size: "1024x1024")
  → 生成された透過PNG を R2 に保存 → designFileUrl に設定
  → /api/printful-mockup → Printful Mockup Generator API（T シャツに合成）
  → モックアップ画像を R2 にキャッシュ → designs.mockupUrl に保存
  → デザインステップで「デザイン画像」と「T シャツモックアップ」を並べて表示
```

**公開キャンペーンページ表示:**
```
campaign.design.mockupUrl があれば → T シャツモックアップ画像を表示
なければ → designFileUrl（アップロード画像）を表示（従来動作を維持）
```

**新規 env var:** `OPENAI_API_KEY`（Printful APIキーは既存の `PRINTFUL_API_KEY` を流用）

3. **Stripe Connect テストオンボーディング（ウィザード Step 3）:**
   - 確認コード: `000000`（テスト用）
   - Business type: Unregistered Business、EIN: No
   - Phone: `000 000 0000`、Email: 任意

##### Cloudflare R2（ファイルストレージ）

1. [dash.cloudflare.com](https://dash.cloudflare.com) → R2 Object Storage
2. 「Create bucket」→ バケット名入力（例: `communitymerch-uploads`）、Location: Automatic
3. バケット → Settings タブ → **Public access** → 「Allow Access」→ 表示された **Public development URL** (`https://pub-xxx.r2.dev`) をコピー
4. **Account ID の確認:**
   - ダッシュボード URL に含まれる: `https://dash.cloudflare.com/{ACCOUNT_ID}/...`
   - または R2 overview ページ右側に表示される
5. **R2 API トークンの生成（Account API Token を推奨）:**
   - R2 → 右側の「Manage R2 API Tokens」→ 「Create Account API token」
   - Token name: 任意（例: `communitymerch-uploads-rw`）
   - Permissions: **Object Read & Write** を選択
   - Specify bucket(s): **Apply to all buckets in this account**
   - 「Create Account API Token」をクリック
   - 表示された以下を**必ずメモ（一度しか表示されない）:**
     - **Access Key ID**: `cfat_...` ではなく S3互換の `xxxxxxx`（32文字）
     - **Secret Access Key**: 64文字の文字列
6. 取得した値を `.env.local` に設定:
   ```bash
   CLOUDFLARE_R2_ACCOUNT_ID=（URL から取得した32文字）
   CLOUDFLARE_R2_ACCESS_KEY_ID=（Access Key ID）
   CLOUDFLARE_R2_SECRET_ACCESS_KEY=（Secret Access Key）
   CLOUDFLARE_R2_BUCKET_NAME=communitymerch-uploads
   CLOUDFLARE_R2_PUBLIC_URL=https://pub-xxx.r2.dev
   ```
**注意:** S3 API 認証には「Token value（`cfat_...`）」ではなく、「**S3 clients 用の Access Key ID と Secret Access Key**」を使用すること（同じ画面に両方表示される）

##### Printful（Print-on-Demand）

1. [printful.com](https://www.printful.com) → アカウント作成（個人メールで可）
2. Dashboard → Stores → 「Connect via API」→ ストア名入力（例: `Community Merch Platform`）→ Continue
3. Settings → Store settings → API → 「Go to Printful Developers」
4. [developers.printful.com](https://developers.printful.com) → 「Sign in to Printful」でログイン
5. Tokens → 「+」(Add new token) をクリック
6. フォームを入力:
   - Token name: 任意（例: `Community Merch Platform Dev`）
   - Contact email: 自分のメールアドレス
   - Expiration date: カレンダーから最大2年先の日付を選択（必須）
   - Access level: A single store → 作成したストアを選択
   - Scopes: **「View and manage orders of the authorized store」にチェック**
7. 「Create new token」→ 表示されたトークンをコピー（**一度しか表示されない**）
8. `.env.local` に `PRINTFUL_API_KEY=<token>` を設定
9. 認証方式: `Authorization: Bearer <token>` ヘッダー（API v1）
